# Translations — status and scope

## ⚠️ AI-generated, not yet reviewed by a native speaker

`fr.json`, `es.json`, `ht.json`, `de.json`, `pt.json`, and `vi.json` were generated by an LLM
translation pass from `en.json` (the source of truth). They aim for fluent, natural phrasing
rather than literal word-for-word translation, and use one consistent term per concept across
the whole app (e.g. "Apply Now" is translated the same way everywhere it appears).

**They have not been reviewed by a native speaker of any of these languages.** Treat them as a
solid first draft, not verified-accurate — especially before using this in a real job-platform
context where an awkward or incorrect phrase could look unprofessional to candidates or
employers. Recommended next step: have a native speaker of each language review their file
before this ships broadly, particularly `ht.json` (Haitian Creole), which has the least
standardized online tooling/reference material of the six.

## Scope — UI strings only

These files translate **UI chrome only**: navigation, buttons, labels, static page copy, form
labels, error messages. They deliberately do **not** cover:

- **AI-generated content** — resumes, cover letters, and Career Coach output stay in English
  regardless of the UI language. Translating AI output is a separate, future lot (it has
  different quality/accuracy stakes than UI chrome — a mistranslated resume could actually harm
  a candidate's job search).
- **User-generated content** — company reviews, application messages, and profile text a
  candidate wrote themselves. Never auto-translated; showing a candidate's own words back to
  them (or to an employer) in a machine-translated form would misrepresent what they actually
  wrote.

## Coverage (as of the 3rd i18n lot)

This was a phased rollout across three lots; all originally-scoped UI-string work is now done:

- **Lot 1**: infrastructure (cookie-based locale, English-fallback merge, language switcher,
  browser-language detection) + `common`, `nav`, `footer`, `auth`, `home`, `jobs`.
- **Lot 2**: `candidate` (dashboard), `recruiter` (dashboard, Post a Job modal, application
  status), `resumeBuilder`, `coverLetter`, `careerCoach`, `analytics`, `profile`, and full
  translations of `legal.privacy` / `legal.terms`.
- **Lot 3**: `errors` — server action and API route error messages (signup/signin, contact form,
  Career Coach/Resume Analysis/Cover Letter rate limits and premium gates, profile/review
  validation, resume/cover-letter export) — plus `ApplyModal`, a component missed by the first
  two lots.

`en.json` currently has ~605 keys across 15 namespaces, mirrored exactly in all 6 translations
(key parity is checked after every lot).

**Deliberately still English** (not gaps, permanent scope boundaries):
- AI-generated content and user-generated content (see above).
- Score-breakdown category labels in the Resume Builder / Cover Letter score cards (`keywords`,
  `formatting`, etc.) — these are the AI response's own fixed schema field names.
- Dynamic third-party error messages (raw Stripe SDK errors, Supabase auth error text, AI-provider
  config errors) — translating a string we don't author and that varies per failure isn't
  reliable in the way a literal we control is.
- No custom 404/500 page exists in this app (Next.js defaults are used), so there was nothing to
  extract there.

If a hardcoded English string is ever found outside these categories, it's a real gap — add it
following the steps below, it isn't an intentional exception.

## Adding a new key

1. Add the English string to `en.json` under the right namespace.
2. Add the same key to all 6 other locale files — a missing key falls back to English rather
   than showing a raw key like `nav.dashboard`, so this can be done incrementally, but don't
   leave a language out for long.
3. Use `useTranslations('namespace')` (client components) or `getTranslations('namespace')`
   (server components/actions) from `next-intl`, not a hardcoded string.
